日々の言葉アプリ
生成・管理・閲覧の全体像

参考資料の「左手＝制度／右手＝実践」の構成を、このアプリでは 「管理者が作る側」と「閲覧ユーザーが読む側」に置き換え、 中央にGeminiとNeonを橋として配置した説明図です。

読み方：左手＝作る側、中央＝生成と保存の橋、右手＝読む側

左手：作る側

管理者がログインし、テーマ・カテゴリ・文章・参考リンクを追加、編集、削除します。

  • 入口：/admin/login
  • 操作：/admin のServer Actions
  • 権限：daily_words_sessions で管理者判定
中央の橋：Gemini + Neon

Geminiが文章を作り、Neonが正本として保存します。

  • 生成：GEMINI_API_KEY
  • 保存：DATABASE_URL
  • 公開：Vercel Environment Variables
右手：読む側

閲覧ユーザーはログイン不要で、DBに保存された文章を4ペインで読みます。

  • Pane 1：テーマ
  • Pane 2：カテゴリ
  • Pane 3：文章一覧
  • Pane 4：文章詳細・参考リンク

工夫点

5 items
  1. 01

    Gemini APIキーはクライアントへ渡さず、Server Actionの中だけで使用。

  2. 02

    テーマごとに生成プロンプトをDB保存し、カテゴリ選択だけで文章生成できる設計。

  3. 03

    Geminiが壊れたJSONを返しても、プレーンテキストとして保存できるフォールバックを追加。

  4. 04

    Base UIのuncontrolled warningを避けるため、defaultValue変更時にInputを再マウント。

  5. 05

    DB変更はmigration化し、Vercelとローカルで同じスキーマへ揃えられる形に整理。

苦戦した点

5 items
  1. 01

    GitHub pushはパスワード認証が使えず、token認証が必要だった。

  2. 02

    VercelのOutput Directoryがpublicになっており、Next.jsビルド成功後に失敗した。

  3. 03

    .envが空だったため、最初はDATABASE_URLを読み込めなかった。

  4. 04

    Neon SDKの戻り値型が広く、クエリ結果を明示的に行配列として扱う必要があった。

  5. 05

    Geminiの返答が途中で切れたJSONになり、JSON.parseだけでは運用に耐えなかった。

Vercelで忘れやすい設定

環境変数

DATABASE_URL と GEMINI_API_KEY は必須。管理者seedを使うなら ADMIN_* も設定します。

DB migration

ローカルでは npm run db:migrate を実行済み。新しいNeon環境では同じmigrationを適用します。

Output Directory

Next.jsとして扱うため、VercelのOutput Directoryは空欄にします。
